LEAP: Supercharging LLMs for Formal Mathematics with Agentic Frameworks

这篇论文的核心判断是：通用 LLM 的形式化数学瓶颈不只是“不懂 Lean”，而是很难一次性生成长而正确的证明。 LEAP 用 Lean compiler、informal blueprint、AND-OR DAG memory 和 LLM reviewer 组成 agentic proving loop， 让 Gemini-3.1-pro 在 Putnam 2025 上完成 12/12，并在 Lean-IMO-Bench 上显著超过 one-shot formalization。

1. 论文速览 / TL;DR

LEAP 是一个面向 Lean 4 的 agentic theorem proving framework。它不训练专用 prover，而是利用通用 foundation model 的 informal reasoning、instruction following、tool use 和 iterative self-refinement，把证明构造拆成“自然语言计划 → Lean sketch → compiler verification → DAG memory → reviewer-guided backtracking”的循环。

一句话 thesis

对形式化数学来说，关键不是让模型一次性写出完整 Lean proof，而是让模型在一个带 verifier、memory、reviewer 的环境中持续构造和修正 proof DAG；当 accepted sketch 满足 $l_1 \land l_2 \land \cdots \land l_k \Rightarrow g$，并且每个 child lemma 最终被 Lean 证明后，parent goal $g$ 才被关闭。

2. 研究动机

现代 LLM 在 informal math reasoning 上已经很强，但自然语言证明有两个硬问题：逻辑漏洞很难自动发现，长证明的人工 peer review 成本很高。 论文用 Kepler conjecture 的审稿与后续 formal verification 作为例子，说明“能写出看似合理的证明”和“能被机器核验”之间仍有巨大距离。

Formal mathematics 把正确性判定交给 Lean / Isabelle / Coq / HOL Light 这样的 proof assistant，理论上能给出 kernel-level 的严格验证。 但现实瓶颈是：LLM 生成的 Lean proof 常常在 syntax、type、library lemma、proof state 管理上失败，尤其是长、多步、非 routine 的 olympiad-level proof。

论文挑战的默认假设

许多前沿系统依赖 specialized prover model 或 formal corpus fine-tuning，例如 AlphaProof、DeepSeek Prover V2、Seed Prover、Goedel-Prover-V2。 LEAP 的立场更激进：通用 LLM 虽然 one-shot 很弱，但它具备 informal planning、compiler error interpretation、tool interaction 和 self-refinement； 如果 scaffold 正确，不必把所有能力都压进一个 specialized prover。

为什么需要 Lean-IMO-Bench

MiniF2F、PutnamBench 等 benchmark 已经越来越饱和或覆盖侧重点不同。Lean-IMO-Bench 选择短 statement 但证明高度非 routine 的 IMO-style problems， 更强调复杂分解、长链依赖和跨数学领域泛化。它包含 60 题：Basic 30 题、Advanced 30 题，并由 Lean experts 手工 formalize 和 verify statements。

3. 数学表示、建模与算法流程

LEAP 的核心状态是 AND-OR DAG。OR node 表示一个待证明 theorem / lemma；AND node 表示一种已被 Lean 接受的 decomposition sketch， 它把 parent goal 约化为若干 child subgoals。系统持续搜索这个 DAG，直到 root goal 的某条 AND decomposition 下所有 child OR nodes 都被证明。

符号与对象

符号 / 对象含义在 LEAP 中的角色
$g$Lean theorem 或 lemma statement一个 proof goal，也是 OR node
$p$完整 Lean proof不依赖新 proposed lemmas；Lean 通过后关闭 goal
$s$formal proof sketch证明当前 goal，但允许 newly proposed lemmas 处保留 `sorry`
$L=\{l_i\}_{i=1}^{k}$由 blueprint 生成的 subgoals成为 child OR nodes；全部证明后 parent sketch 生效
$D=(V,E)$AND-OR proof DAG保存 proof progress、dependency、lemma reuse 和 search memory
$R_{\text{Lean}}$Lean compiler / verifier给出 hard correctness signal：type-check 或 error message
$R_{\text{LLM}}$LLM reviewer判断 decomposition 是否真的变简单、相关、可推进

主循环：direct proof 失败后转向 blueprint decomposition

  1. Register root: 输入 Lean statement，作为 root OR node 写入 DAG。
  2. State reader: 读取当前 goal、已有依赖、相关 lemmas 和 graph context。
  3. Direct formalization: 先生成 informal proof，再翻译成 Lean proof；失败时用 compiler feedback 与 LeanSearch retrieval 做 revision。
  4. Blueprint generation: direct path 失败后，生成 informal blueprint，提出 intermediate lemmas，并转成 formal sketch。
  5. Sketch verification: Lean 检查 sketch；main theorem body 必须 sorry-free，`sorry` 只允许出现在 newly proposed lemma statements。
  6. LLM review: reviewer 判断 proposed subgoals 是否相关、是否更简单、是否构成 plausible route。
  7. State writer: 通过 acyclicity check 后，把 accepted sketch 写成 AND node，把 subgoals 写成 child OR nodes。
  8. DFS with backtracking: 递归处理 child goals；若 branch 无进展，回溯并尝试 alternative blueprint。

为什么 DAG memory 重要

如果每次 decomposition 都形成独立 tree，重复 subproblem 会被反复发现和证明。DAG 的 memoization 让相同或可复用的 intermediate lemma 成为共享节点， 从而把多个 proof branches 汇聚到同一依赖。作者强调两个效果：monotone refinement 保持全局 proof plan 的稳定依赖顺序； anticipatory lemma planning 允许上层 blueprint 提前提出后续可能有用的 auxiliary lemmas。

可验证性的关键不变量： 一个 decomposition 只有在 Lean 已经验证“当前 goal 可由 proposed lemmas 推出”时才进入 DAG。 因此 LEAP 的 search 可以失败、回溯、浪费 compute，但不会把未经 Lean 接受的 proof 当作 solved proof。

形式化语义的直观表达

设当前 goal 为 $g$，blueprint 提出 subgoals $l_1,\ldots,l_k$。LEAP 不是直接相信自然语言计划，而是要求 Lean 接受一个 sketch：

$$\text{LeanAccept}\left(s: l_1,\ldots,l_k \vdash g\right)=1.$$

当 $l_1,\ldots,l_k$ 后续分别被完整 Lean proof 关闭时，DAG 中对应 AND node 才能关闭 parent $g$。这使 natural-language blueprint 只负责提出方向， correctness 仍由 Lean kernel 兜底。

7. 复现清单

复现难度为 medium-to-high：Lean proof artifacts 和 benchmark 入口公开，但 exact prompts、decoding、hardware、timeout、model access 没有完整披露。 如果只能访问不同版本的 Gemini 或开源 LLM，结果很可能无法直接对齐论文数字。

必需资源

  • Lean 4 与 Mathlib 环境。
  • LEAP Lean solutions / code: `google-deepmind/superhuman/tree/main/leap`。
  • Lean-IMO-Bench statements: `https://imobench.github.io`。
  • LeanSearch retrieval，论文 Figure 1 中 direct / blueprint paths 都连接了 Lean Search。
  • 可调用 Gemini-3.1-pro 或可替代的 general-purpose LLM。

实验日志要记录

  • 每题 LLM calls、active DAG nodes、最终 proof line count。
  • 每个 accepted sketch 的 proposed lemmas、Lean errors、revision history。
  • LLM reviewer 的 reject reasons 与 backtracking branches。
  • Pass@128 sampling config、rollout=2 的 budget 定义、per-problem timeout。
  • 所有最终 `.lean` 文件必须由 Lean kernel 重新验证。

最小复现路线

  1. 先复现 Table 5 的 Direct Formalization：对 Lean-IMO-Bench Basic set 跑 one-shot Pass@128，再跑最多 20 轮 compiler feedback revision。
  2. 实现 state reader / writer，把 root theorem 注册为 OR node，保存 dependencies、proof attempts 和 compiler messages。
  3. 实现 direct path：informal proof generation → Lean code generation → Lean verify → revise。
  4. 实现 decomposition path：informal blueprint → formal sketch → Lean verify sketch；只允许 proposed lemma 处保留 `sorry`。
  5. 加入 LLM reviewer，拒绝与 parent goal 等价、过难、无关或无法推进的 subgoals。
  6. 加入 DAG memoization 与 acyclicity check，再复现 Table 6 的 Naive Tree vs Full DAG。
  7. 最后扩展到 Putnam 2025，按 Table 3 记录 calls / nodes / proof length。

8. Reviewer Critique

优点

  • 问题定位准确： 把 formal proving 的难点从“模型是否会数学”细化为“长 proof 的 decomposition、state management、compiler feedback use”。
  • 系统设计有可验证不变量： accepted sketch 必须通过 Lean，避免纯自然语言 reviewer 直接决定 correctness。
  • 证据强： Putnam 2025 逐题、Lean-IMO-Bench 分领域、DAG ablation、LLM reviewer failure case 都支撑主要 claim。
  • benchmark 有价值： Lean-IMO-Bench 填补短 statement、长非 routine proof 的评测空缺。

主要弱点与威胁

  • 复现细节不足： prompt、decoding、hardware、timeout、reviewer criteria、branch priority 都不够完整；对 agentic system 来说这些会显著影响结果。
  • compute 可比性不足： Pass@128、rollout=2、7-day Hilbert cap 和 LEAP 的 per-problem budget 并不完全等价。
  • backend 依赖强： Gemini-3.1-pro 的可用性和版本差异会影响 external reproducibility。
  • Geometry 弱项明显： Lean-IMO-Bench Geometry 几乎仍未解决，说明 framework 不能替代 domain-specific libraries / tactics。
  • LLM reviewer 仍可能误判： Figure 3 证明 reviewer 很重要，但还没有大规模 reviewer precision / recall 或错误 taxonomy。

我会要求补的实验

  • 在更多 Putnam / Lean-IMO problems 上系统移除 LLM reviewer，而不只聚焦 A5。
  • 报告 Full DAG 相比 Naive Tree 的 LLM call reduction、wall-clock、重复 lemma reuse rate。
  • 做 model swap：同一 LEAP scaffold 下比较 Gemini、OpenAI、Claude、开源 reasoning model。
  • 给出 prompt templates、sampling params、branch budget 和 failure logs，以便第三方复现。
  • 尝试 hybrid architecture：foundation model 做 planner/reviewer，specialized prover 做 local Lean proof generation。

9. 对我的研究方向的启发

LEAP 最值得迁移的不是“Lean 专用技巧”，而是 verifier-grounded agent design：把任务拆成可验证子目标，用 graph memory 保存中间状态， 再让 reviewer 判断“是否真的推进了问题”。

RLVR / self-evolution

Lean compiler 给 binary reward，但 DAG 中的 accepted sketch、open-goal reduction、lemma reuse 可作为 process reward。失败 traces 可进入 replay buffer 训练 reviewer 或 proposer。

Questioner-solver 分工

Blueprint generator 像 questioner，提出 subgoals；formal prover 像 solver；LLM reviewer / Lean verifier 组成 evaluator。这个三方结构比单模型 self-consistency 更可控。

Diversity 不是盲采样

OR nodes 允许多种 proof strategy，但 DAG memoization 把有用 lemmas 汇聚复用。多样性应该围绕 reusable subgoals，而不是只增加 complete proof samples。

Evaluation design

Lean-IMO-Bench 的启发是：短 statement、长 proof、hard-to-search 的任务更能区分 scaffold 能力。对 agent benchmark 来说，应避免只测容易被 one-shot 或 memorization 覆盖的任务。

最终判断

这是一篇值得精读的 agentic formal reasoning paper。它的强 claim 主要成立：通用 LLM 在 verifier-guided scaffold 中能显著缩小与 specialized prover 的差距。 但结论不应解读为 specialized prover 不重要；更可能的下一步是 hybrid system，把 LEAP 的 global planning / reviewer / memory 与 specialized local prover 结合。
